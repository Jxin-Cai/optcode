# 代码风格一致性审查视角

## 检查清单

1. **命名规范** — 变量、函数、类、文件的命名风格不一致
   - 同一项目混用 camelCase 和 snake_case
   - 缩写不一致（如 ID vs Id vs id）
   - 布尔变量缺少 is/has/should 前缀
   - 函数命名不符合语言惯例

2. **缩进和格式** — 缩进风格不一致
   - 混用 tab 和空格
   - 缩进层级不一致
   - 大括号风格不一致（如 K&R vs Allman）

3. **注释风格** — 注释不规范
   - 过期或不准确的注释
   - 注释风格不一致
   - TODO/FIXME/HACK 注释积压

4. **文件组织** — 文件内部结构不一致
   - import 顺序不一致
   - 类成员排列顺序不一致
   - 文件长度过长未合理拆分

5. **空行和间距** — 代码块间的空行使用不一致
   - 函数间空行数量不一致
   - 逻辑块间缺少适当的空行分隔

## 严重度判定

- **high**: 命名严重误导（如布尔值命名暗示相反含义）、格式混乱到影响代码审阅
- **medium**: 同一项目中风格不一致（如混用命名约定）、过期的误导性注释
- **low**: 细微的格式偏差、空行不一致

## 维度规则

- 以项目自身的主流风格为基准，不强加外部标准。先统计项目中的主流风格，再报告偏离主流的代码
- 如果项目有 linter/formatter 配置文件（如 .eslintrc、.prettierrc、.editorconfig、gofmt），以配置为准
- 只报告影响可读性或一致性的问题，不纠结于无关紧要的格式差异
- 修复方案优先建议使用项目已有的格式化工具自动修复
- 不要报告已被 .editorconfig 或格式化工具覆盖的问题
- 修复风险判定：格式化/空行调整/import 排序 = safe；重命名局部变量 = safe；重命名导出的函数/类/常量 = local（需更新所有调用方）
