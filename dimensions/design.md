# 设计原则审查视角

## 检查清单

1. **单一职责原则（SRP）** — 类/模块/函数承担了过多的职责
   - 一个类超过 300 行且包含多个不相关的功能域
   - 一个函数做了多件不相关的事情
   - 一个模块混合了不同层次的关注点（如业务逻辑混合 IO 操作）

2. **开闭原则（OCP）** — 修改现有代码才能扩展行为
   - 大量 if/else 或 switch 来区分类型，添加新类型需要修改已有代码
   - 缺少扩展点，每次新增功能都要改核心逻辑

3. **高内聚低耦合** — 模块间耦合过紧或模块内聚度过低
   - 模块间存在循环依赖
   - 大量跨模块直接访问内部实现（而非通过接口/API）
   - 一个变更需要同时修改多个不相关模块

4. **依赖倒置原则（DIP）** — 高层模块直接依赖低层实现
   - 业务逻辑直接依赖具体的数据库/网络/文件实现

5. **接口隔离原则（ISP）** — 接口过于庞大
   - 一个接口/抽象类包含太多方法，实现类被迫实现不需要的方法

6. **分层边界** — 架构分层被穿透或不存在
   - 业务逻辑直接调用 HTTP/数据库/文件系统等 IO 操作，未通过抽象层隔离
   - 表现层直接操作数据存储，跳过业务层
   - 跨层调用（如 controller 直接调 DAO）

7. **全局状态与副作用** — 难以追踪的隐式状态
   - 可变全局变量被多处读写
   - 函数通过修改外部状态而非返回值来传递结果
   - 初始化顺序依赖（模块 A 必须在模块 B 之前初始化，但无显式约束）

8. **框架耦合** — 核心逻辑与特定框架深度绑定
   - 业务逻辑类继承框架基类或实现框架接口
   - 业务函数中直接使用框架特定 API（如 ORM 查询语法散落在业务逻辑中）

## 严重度判定

- **high**: 循环依赖、业务逻辑与 IO 混杂导致无法独立修改、全局可变状态被多处并发读写、违反导致新增功能必须改核心代码
- **medium**: 模块间耦合过紧但当前尚可维护、框架耦合度高但替换需求不紧迫、接口过大但实现类不多
- **low**: 设计不够优雅但不影响日常开发效率、轻微的分层穿透

## 维度规则

- 只报告确实造成问题的设计违反，不追求教科书式的完美设计
- 评估时考虑项目规模和复杂度——小型项目或脚本不需要过度设计
- 修复方案必须在当前架构下可行，不要建议推翻重写
- 必须说明为什么这个设计违反在此处是个问题（而不只是引用原则名称）
- 对于已成型的架构模式（即使不完美），如果改动成本远大于收益，标注但不强制修复
- 修复风险判定：提取常量/简化条件 = safe；提取接口/拆分函数 = local；拆分模块/调整依赖方向/引入抽象层 = structural；改变公共 API 契约 = behavior-risk
