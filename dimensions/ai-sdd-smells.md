# AI/SDD 坏味道审查视角

## 检查清单

1. **需求漂移** — 实现偏离 spec/任务描述，添加未要求行为
   - 无依据的兼容分支、默认值、fallback
   - 把临时调试或样例逻辑变成正式路径

2. **过度工程** — 为单一场景引入不必要抽象
   - 未被复用的接口、策略、工厂、配置层
   - 为假想未来需求增加复杂分支

3. **AI 自述式修复** — 报告说修复但代码证据不足
   - 测试只覆盖 mock 或实现细节，无法保护行为
   - 注释解释“已修复/用于某任务”，而不是代码本身表达意图

4. **错误处理膨胀** — 捕获不可能发生的内部错误
   - 空 catch、吞异常、泛化错误信息
   - 在内部可信路径加入多层防御导致真实问题被隐藏

5. **上下文污染** — 为满足局部请求引入全局改动
   - 无关文件重构、格式化、重命名
   - 修改公共 API 但没有调用方证据

## 严重度判定

- **high**: 需求漂移影响外部行为、公共 API 被无依据修改、测试无法保护核心行为
- **medium**: 过度抽象增加维护成本、错误处理隐藏真实失败、无关改动扩大变更半径
- **low**: 局部注释/命名带有任务痕迹、单处可删除 fallback

## 维度规则

- 只报告有代码证据的问题，不把“像 AI 写的”作为证据
- 必须指出 spec/调用方/现有模式中的对照证据
- 优先建议删除或收敛复杂度，而不是新增抽象
- 对公共 API、行为语义、测试策略相关问题标为 behavior-risk 或 structural
- 修复方案必须保持上下文克制：小步删除、收敛、补行为验证，不做大重构
