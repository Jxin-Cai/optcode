# 可维护性审查视角

## 检查清单

1. **代码可读性** — 代码是否易于理解
   - 过长的函数（超过 50 行且逻辑复杂）
   - 过深的嵌套（超过 3 层的 if/for/while 嵌套）
   - 复杂的表达式（多个三元运算符嵌套、复杂的布尔表达式）
   - 魔法数字/魔法字符串（未提取为常量的字面量）
   - 非直觉的控制流（多重 break/continue/goto、异常驱动的控制流）

2. **模块化程度** — 代码是否合理模块化
   - 上帝对象/上帝函数（一个函数/类做了太多事情）
   - 不合理的文件划分（一个文件包含多个不相关的类/函数）
   - 紧密耦合的模块（修改一处需要连锁修改多处）

3. **扩展难度** — 代码是否易于扩展
   - 硬编码的配置/阈值/路径（应该抽取为配置）
   - 缺少必要的抽象层（直接面向实现编程）
   - 过于具体化的实现（只适用于当前场景，无法泛化）

4. **错误处理健壮性** — 错误处理是否充分
   - 吞掉异常不处理（空 catch 块）
   - 缺少关键路径的错误处理
   - 错误信息不够具体，无法定位问题

## 严重度判定

- **high**: 核心路径上的上帝函数（>200 行）、嵌套超 5 层的逻辑、吞掉异常的空 catch 块
- **medium**: 50-200 行的复杂函数、魔法数字在多处使用、缺少关键路径错误处理
- **low**: 非核心路径的轻微可读性问题、单处魔法数字

## 维度规则

- 只报告确实影响可维护性的问题，不追求完美代码
- 优先关注高频修改的核心路径——边缘的一次性脚本不必严格要求
- 修复方案必须渐进可行，不要建议大规模重构
- 必须说明为什么这个问题影响可维护性（而不只是"看起来不太好"）
- 区分"应该改"和"可以改"——只报告"应该改"的问题
- 修复风险判定：提取常量/简化表达式/补充错误信息 = safe；拆分长函数/降低嵌套层级 = local；拆分文件/模块重组 = structural；改变错误处理逻辑或控制流 = behavior-risk
