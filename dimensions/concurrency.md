# 并发安全审查视角

## 检查清单

1. **竞态条件** — 多线程/协程/异步操作对共享数据的非同步访问
2. **死锁** — 多个锁的获取顺序不一致导致的潜在死锁
3. **线程不安全的共享资源** — 非线程安全的数据结构在并发环境中使用（如 HashMap 并发写、ArrayList 并发修改）
4. **原子性违反** — check-then-act、read-modify-write 等复合操作缺少同步
5. **可见性问题** — 共享变量缺少 volatile/atomic/锁保护，导致线程间不可见
6. **异步错误处理** — Promise/Future/Channel 的错误未正确传播或处理
7. **资源泄漏** — 并发场景下的连接、锁、文件句柄未正确释放

## 严重度判定

- **high**: 竞态条件、死锁、数据损坏风险——可导致数据不一致或系统挂起
- **medium**: 可见性问题、非最优的同步策略——可能导致间歇性 bug
- **low**: 资源泄漏风险较低的场景、异步错误处理不完善但不影响正确性

## 维度规则

- 先判断代码的并发模型（多线程、协程、异步、多进程），再评估问题
- 必须提供完整的问题触发路径（哪些线程/协程、什么操作顺序、什么共享资源）
- 修复方案必须具体到使用什么同步原语（mutex、channel、atomic、synchronized 等）
- 如果项目是单线程/单进程且没有并发代码，直接输出 pass，不要编造并发问题
- 修复风险判定：添加 volatile/atomic 标注 = safe；添加锁保护局部代码段 = local；重新设计同步策略或改变数据流方向 = structural；改变错误传播语义 = behavior-risk
